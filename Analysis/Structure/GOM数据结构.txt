GOM数据结构

0. 名词
     - GOM: GameObjectManager，Unity 用于按 key 分桶组织对象的容器（bucket + 环形双链表）
     - Bucket: 哈希桶条目，包含 hashMask/key 与链表头
     - ListNode: 桶内链表节点，节点携带 nativeObject 指针
     - listHead: 环形双链表的“哨兵节点”或“头节点”（实现按版本可能等价于某个 ListNode）

1. 全局入口（定位 GOM）
     - module: UnityPlayer.dll
     - symbol: UnityPlayer.dll + GameObjectManagerOffset -> gomGlobal（全局指针槽地址）
     - manager = *(gomGlobal) -> GameObjectManager*
     - GameObjectManagerOffset 可手动逆向定位，也可通过运行时盲扫自动获取

2. GameObjectManager（结构语义）
     2.1 字段布局（64-bit）
         - +0x00 buckets: Bucket*，桶数组首地址
         - +0x08 bucketCount: int32，桶数量

     2.2 读取校验点
         - buckets != 0
         - bucketCount 在合理范围（典型：>0 且不会大到导致 buckets*stride 越界）

3. Bucket（stride = 0x18）
     3.1 字段布局
         - +0x00 hashMask: uint32，桶的哈希掩码（用于快速筛桶）
         - +0x04 info: uint32，桶附加信息（用途不固定，按实现作为保留/前缀/标记）
         - +0x08 key: uint64，桶键（常用于 tagId 等）
         - +0x10 listHead: ListNode*，桶内环形链表头

     3.2 不变量
         - listHead != 0 才认为桶有效；无节点时可能为 0 或指向自环哨兵
         - 同一桶内所有节点属于同一 key/hashMask 分组

4. ListNode（stride = 0x18，环形双链表）
     4.1 字段布局
         - +0x00 pLast: ListNode*，前驱
         - +0x08 pNext: ListNode*，后继
         - +0x10 nativeObject: NativeGameObject*（原生对象指针）

     4.2 结构约束
         - 节点为环形链：从任意节点沿 pNext 迭代，最终回到起点
         - 遍历必须设置安全上限（防止链表损坏/读失败导致死循环）

5. NativeGameObject（只描述与 GOM 关联的桥接点）
     - 该指针用于进一步解析 managedObject、name、components 等（具体布局见 IL2CPP/MONO 内存结构文档）

6. 版本差异提示（仅影响“如何验证桶类型”，不影响上述字段偏移）
     - 旧实现可能有“manager + 固定偏移 -> 某个 list”的走法；新实现为 bucket 结构
     - 是否需要对 bucket 做类型过滤（例如只保留 GameObject 桶）取决于上层枚举策略
